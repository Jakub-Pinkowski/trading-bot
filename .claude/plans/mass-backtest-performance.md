### MACD run vs RSI run — key differences

**The averages move in opposite directions:**

| Strategy | Start avg | At ~24k tests | Trend       |
|----------|-----------|---------------|-------------|
| RSI      | 3.81s     | 4.66s         | ↑ worsening |
| MACD     | 5.19s     | 3.67s         | ↓ improving |

**Why MACD improves while RSI degraded:**

MACD with 5 fast × 5 slow × 5 signal × 2 rollover × 7 trailing × 2 slippage =
**3,500 strategies per `(symbol, interval)` group** vs RSI's 252.

Each group takes ~35 batches to complete vs RSI's ~2.5. This completely changes the economics:

- **Cold-start tax** (loading parquet + computing 125 unique MACD indicator combinations) is
  paid once per group and amortized over 35 batches of hot-cache computation.
- Workers stay within a single `(symbol, interval)` group for long enough that cross-group
  cache pollution is rare relative to total tests.
- The "fast zone" at batches 106–140 (0.78–1.35s batches) is a single group running fully
  warm — the indicator cache is loaded, workers just iterate through parameter variants.

RSI groups were too small (252 = 2.5 batches) for this amortization to help. Workers moved to
new groups fast enough that memory from all groups accumulated simultaneously.

**The 15.52s spike at batch 2000** is the combination of: an intermediate save on a growing
parquet + a group transition where all 8 workers simultaneously encounter a new
`(symbol, interval)` with 125 uncached MACD indicator combinations to compute.

**What this means for the end of the MACD run (154k tests, ~16× more than RSI):**

We only have data for 16% of the run. The declining average is encouraging, but the RSI
experience showed the floor breaks at a certain scale. With 154k tests, the later symbols
could still trigger the same memory-pressure spiral seen in RSI, just delayed. Impossible to
tell without more data.

**The O(N²) save problem is now much more severe.** The MACD parquet file will eventually hold
~154k rows. At the last save (test 154,000), `save_to_parquet` will read 153,000 rows,
concatenate, dedup, and rewrite. This will take significantly longer than anything seen in the
RSI run.

## Root Cause Summary

Two independent problems drive the variability and slowdown observed in the logs:

1. **Unbounded per-worker memory growth.** 8 worker processes each maintain their own
   `dataframe_cache` (max 50 DataFrames) and `indicator_cache` (max 500 Series). These are
   module-level singletons that accumulate across the entire run. By the time sym5–sym7 are
   processed (~4000–6000 tests), each worker holds multiple large DataFrames and Series in
   memory simultaneously, pushing total RSS well above physical RAM and causing OS paging.

2. **O(N²) intermediate saves.** `save_results()` at every 1000 tests reads the entire
   existing parquet file, concatenates, deduplicates, and rewrites it. As the file grows the
   cost of each save increases linearly, visible as the consistent post-save spikes.

The per-symbol cold-cache sawtooth (fast batches interleaved with slow ones) is expected and
unavoidable; the proposals below focus on the two fixable problems above.

## Proposals (ranked by impact)

---

### 1. Chunked execution — process one `(symbol, interval)` group at a time

**Impact: Very High | Effort: Medium**

**The problem:** All 9504 tasks are submitted to the executor at once. With 8 workers each
holding up to 50 cached DataFrames, peak memory scales with the number of distinct
`(symbol, interval)` pairs that happen to be warm simultaneously (up to 8 × 50 = 400
DataFrames). Nothing bounds this.

**The fix:** Restructure `_execute_tests_in_parallel` to submit one `(symbol, interval)` chunk
at a time (252 tasks for RSI) instead of all tasks upfront. After each chunk completes, the
workers' caches contain at most one DataFrame and one RSI Series each — then those entries age
out naturally on the next chunk.

```python
# Pseudocode — group by (symbol, interval) before submitting
chunks = group_by_symbol_interval(test_combinations)
for chunk in chunks:
    futures = {executor.submit(run_single_test, p): p for p in chunk}
    for future in as_completed(futures):
        ...
```

**Why it works:** For RSI with a fixed period=13, each `(symbol, interval)` group requires
exactly one DataFrame load and one RSI computation. All 252 strategy variants (thresholds,
rollovers, trailing stops, slippage) reuse those two cached objects. After the chunk finishes,
no other chunk needs those objects, so they can be evicted.

**Peak memory becomes:** 8 workers × 1 DataFrame × 1 RSI Series — constant regardless of
total symbols.

### 2. Fix the O(N²) intermediate save pattern

**Impact: High | Effort: Low**

**The problem:** `save_results()` in `reporting.py` → `save_to_parquet()`:

1. Reads the entire existing parquet file into memory
2. Concatenates with new results
3. Drops duplicates across the full combined set
4. Rewrites the entire file

At 1000 tests saved, this reads ~1000 rows. At 5000 saved tests (6th save), it reads ~5000
rows and rewrites them all. This gets worse with each call.

**The fix:** Two options:

**Option A — Append-only intermediates, merge at end.**
Write intermediate results to numbered shard files
(`mass_test_results_shard_001.parquet`, etc.) and merge + deduplicate only once at the end of
the run.

```python
# On intermediate save (every 1000 tests):
shard_path = f"{BACKTESTING_DIR}/shards/shard_{shard_index:04d}.parquet"
results_df.to_parquet(shard_path, index=False)

# At end of run:
all_shards = pd.concat([pd.read_parquet(p) for p in shard_paths])
all_shards.drop_duplicates().to_parquet(final_path, index=False)
```

**Option B — Remove deduplication from intermediate saves.**
Deduplication is only needed because `skip_existing=False` re-runs could produce duplicates.
When `skip_existing=True` (the normal case), rows are inherently unique. Remove the
read+concat+dedup from `save_to_parquet` for intermediate saves; keep it only for the final
write.

Option A is safer and avoids ever loading the full dataset mid-run.

---

### 3. Lower the worker cache max sizes

**Impact: Medium | Effort: Very Low**

**The problem:** `dataframe_cache = Cache("dataframe", max_size=50, ...)` and
`indicator_cache = Cache("indicator", max_size=500, ...)`. These are defined in module-level
singletons, so each of 8 workers can hold up to 50 DataFrames and 500 Series. For the RSI-only
run with 9 symbols × 4 intervals, a worker will accumulate up to 36 DataFrames and 36 RSI
Series before LRU eviction kicks in at all.

**The fix:** Reduce the max sizes:

- `dataframe_cache`: reduce from 50 to ~12 (enough to keep the current and previous couple of
  symbol/interval pairs warm, but not the entire run)
- `indicator_cache`: reduce from 500 to ~50 (RSI-only run only generates 36 unique entries)

```python
# dataframe_cache.py
dataframe_cache = Cache("dataframe", max_size=12, max_age=604800)

# indicators_cache.py
indicator_cache = Cache("indicator", max_size=50, max_age=2592000)
```

**Tradeoff:** If future runs add more strategies or more intervals, a too-small cache would
increase cache misses. But the current sizes are far larger than needed for any single run.

### 4. Pre-compute indicators in the main process and pass results to workers

**Impact: Medium | Effort: High**

**The problem:** Each worker independently computes RSI for every `(symbol, interval)` pair it
encounters. With 8 workers and no shared memory, the same RSI Series is computed up to 8 times
(once per worker) per `(symbol, interval)`.

**The fix:** In the orchestrator, before the executor loop, pre-load each DataFrame and
pre-compute its indicators. Store them in a dict keyed by `(symbol, interval)`. Pass the
pre-computed indicator Series as part of the task params. Workers skip the indicator
computation entirely.

```python
# In orchestrator, before executor:
precomputed = {}
for symbol, interval in unique_pairs:
    df = pd.read_parquet(filepath)
    precomputed[(symbol, interval)] = {
        'df': df,
        'rsi_13': calculate_rsi(df['close'], period=13, ...)
    }

# Add to task params, workers use directly
```

**Tradeoff:** Pre-computed data is serialized and sent to workers via pickle over IPC. For
large DataFrames this adds overhead. This proposal is best combined with Proposal 1 (chunked
execution), where you only pre-compute the current chunk's data rather than everything at once.
For shared memory the ideal solution is Python 3.8+ `multiprocessing.shared_memory`, which
avoids copying entirely, but requires careful lifecycle management.

### 5. Move the batch progress timer to exclude the main-thread save cost

**Impact: Low | Effort: Very Low**

**The problem:** The "Batch" timer in `_execute_tests_in_parallel` includes time spent in
`save_results()` when `completed_tests % 1000 == 0`. The save runs in the main thread before
the next `as_completed` iteration, making the batch that hits the 1000-count boundary appear
artificially slow.

**The fix:** Reset `batch_start_time` after the save, not before:

```python
if completed_tests % 1000 == 0 and tester.results:
    save_results(tester.results)
    tester.results.clear()
    batch_start_time = time.time()  # reset AFTER save so it doesn't pollute batch timing
```

This is a metrics fix, not a perf fix — but it removes misleading noise from the progress
output and makes the other proposals easier to evaluate.

---

### 6. Fix the cache statistics reporting bug

**Impact: Low (correctness) | Effort: Very Low**

**The problem:** `_report_cache_statistics` iterates over `tester.results`, but `tester.results`
is cleared every 1000 tests (orchestrator.py:279). By the time reporting runs at the end, only
the last ~504 results are present. The stats (978 hits / 16 misses) represent only ~5% of the
actual run. Additionally, `indicator_cache.size()` and `dataframe_cache.size()` report the
**main process's** cache size, which is always 0 — workers run in separate processes with their
own instances.

**The fix:** Accumulate cache stats as a running total alongside results, not derived from the
results list after it has been cleared.

```python
# In _execute_tests_in_parallel, maintain running counters:
total_cache_stats = {'ind_hits': 0, 'ind_misses': 0, 'df_hits': 0, 'df_misses': 0}

for future in as_completed(future_to_test):
    result = future.result()
    if result and 'cache_stats' in result:
        for k in total_cache_stats:
            total_cache_stats[k] += result['cache_stats'][k]
    ...
    if completed_tests % 1000 == 0:
        save_results(tester.results)
        tester.results.clear()
        # Do NOT clear total_cache_stats — it's cumulative

# Pass total_cache_stats to _report_cache_statistics instead of tester.results
```

**Revised priority based on MACD data:**

Proposal 2 (saves) is now the single most important fix. For MACD it is not O(N²) but
O(N·M) where M = file row count — which approaches 154,000 by the final save. The last save
alone will read+rewrite the entire dataset. At the rate saves are happening (every 1000 tests
= every ~40s early on), this will become the dominant bottleneck before the run finishes.

Proposal 1 (chunked execution) is still the correct long-term solution for RSI-type runs
where small group sizes cause continuous memory pressure. For MACD, its main value is
insurance against the unknown late-run trajectory (we only have 16% of the data) and making
cold-start spikes predictable rather than random.

Proposal 4 (pre-compute indicators) is **not recommended for MACD**. MACD has 125 unique
indicator combinations per group; serializing those via IPC to 8 workers would likely be
slower than letting workers compute them from the warm-cache disk files. Stick to the current
model for indicator-heavy strategies.
